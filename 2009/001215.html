I have no explanation, so far, as to why UUIDs would be different.
The most obvious conjecture is not that the UUIDs in fstab were incorrect,
but rather that at some time the UUIDs of those filesystems changed.
pvcreate creates UUIDs for physical volumes.
However, does creating a filesystem on that volume
then also change the UUID?
And if so, what filesystem?
I want to check this out please.
What filesystems were you using, please?

Because the UUID of a filesystem is read
and then stored in array PUUID
it is possible that it could change
and have not been read again.

Perhaps repartitioning a disk,
then reloading might have caused the
stored UUID values to become incorrect
or in the wrong place?

> On Sat, May 23, 2009 at 9:03 AM, Kyle Sallee <kyle.sallee at gmail.com> wrote:
>> I made it so that sorcery does not invoke the regeneration scripts in
>> /etc/init.d/
>> automatically from cast, while casting when booted from the I/R.
>> Perhaps in the past, a SA too anxious to complete casting might have
>> ended the chroot too soon and that might have been the reason why
>> some SAs ended up with boxes where lilo was not installed or
>> the initramfs was not properly regenerated.
>>
>> Now, following the completion of cast during the installation
>> the regeneration scripts are invoked directly from the installer
>> and run in the foreground so that the completion of those tasks
>> are waited for before ending the chroot and unmounting
>> virtual filesystems that were bound into the install filesystem.
>> With this change proper symlink generation in /etc/rc.d/rc*.d/
>> will be correct, the initramfs will be correct, and lilo will have
>> been installed.
>>
>> I apologize for not having realized that as a possible potential for
>> causing a failed installation previously.
>>
