I used to have the sorcery default to optimization
-march=native -O,
however I recently changed that to -march=native -Os.
The I/Rs have become larger than expected,
and that may in part be due to the fact that
a long time ago I used -Os for the I/Rs.

Software that has a tendency to run full bore,
such as bash ( when used by sorcery ),
bzip2, gzip, 7za , lzip, etc... still receive the -O3
optimization instead of the -O2 optimization.
Even MesaLib receives -O3 optimization,
because I realize some SA want the
fastest possible 3D graphics.

The reason I did not want to use -O2 or -O3 for defaults
was because they contain optimizations that
require longer for compilation.
And gcc is does not become any faster with each new release.

Also SAs can still set up a different default cpref,
or they can specify different cpref for spells.

Also -O3 is not guaranteed to produce code that runs faster than -Os.
For default -Os may be the faster choice,
especially for programs that run for a short duration and then exit.
In the past sometimes -O3 code segfaults.
And historically the only optimization expected to work
for compiling glibc was -O2 and therefore
glibc is always compiled with- O2 optimization
