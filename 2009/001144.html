Ok.
That is news to me.
Thanks for reporting it.
Since I do not have jdk installed on any 64 bit architectures
I was unaware that it was missing.

It seems unclear why a whole 64 bit jdk would be provided,
yet be without the 64 bit plugin.
Perhaps there are other java implementations we should consider?
Could perhaps gcc's java be used as a plugin?
Nobody is probably eager to try and compile jdk from source,
but it could be attempted if one has a few weeks to try it.


Currently, I am in the process of creating an i586-linux-uclibc
port of Sorcerer.
Of course jdk will not run on it.
Consequently, we could use some alternatives.

For the moment I have a long way to go before the port is complete.
I am doing much manual compilation merely because sentinel is
unusable as a preload in a mixed target environment.
But eventually, I should have enough of it as i586-linux-uclibc
that I can use cast to properly install the software and
create archives.

If sorcery used DESTDIR instead of sentinel
then I would be inclined to provide an easy method
for transitioning deployed boxes from glibc to uClibc.
However, since that is not the situation,
those who want to try the uClibc will have to install
it chrooted, or as a separate installation, or replace
their existing installation with it.

Whatever SAs decide it still will be available for testing and evaluation.
While on the subject of testing and evaluation,


Sorcerer will not be updating to gcc version 4.4.0
It is too buggy.
Even a glibc compiled with gcc version 4.4.0
once installed becomes a constant segfault
when anything is attempted to be run,
yes including the statically linked ldconfig.

The inconvenience of a glibc break requires 10 seconds to fix
when one knows what to do.
However, it is something most SAs want to avoid
which is why I try to test new releases of software such as gcc.

My confidence in the quality of new glibc and gcc is slipping.
Many SAs want to call stable software "bleeding edge" but it is not.
Developmental software is "bleeding edge."
Sorcerer has never been a "bleeding edge" distro.
Why?
Because I can not make all that developmental software work.
It is too buggy.
I have a sufficient challenge trying to integrate stable software.

I will try to make updates as close to stable as practical
so that we can enjoy the benefits of bug fixes, security fixes, etc...
However, Sorcerer may not contain as many current stable
releases as it has in the past.
Whether staying stable works or fails is mainly
due to the software authors, and is not my judgment.
But like any SA I will back away from any release that
seems to have a bad stench.

Consequently, when stable releases are made that
do not immediately become part of the stable version array
for a spell in the grimoire
then that version will probably be available as part of the latest
version array.

Selecting version progress for a spell has never been easier.

# augur progress $SPELL $VERSION

$VERSION can be
stable
unstable
latest
or an actual version

When unstable or latest version progress is selected
then the stable version will be used instead of the selected
version progress when selected version progress is
not specified in the spell.

Sometimes spells gain unstable versions and sometimes
the unstable array is removed entirely.
The same can happen for the latest version array.
However, all spells have a stable version array.

gcc did have a latest of 4.4.0
However, I have decided to comment it out,
because we really do not want to be using gcc version 4.4.0

Those who are not used to the new version progress
may yet want to read the augur help screen.
In addition some augur commands have additional
command specific help screens.

# augur help progress

That prints more than 25 lines about it,
hopefully enough to brief the SA on how to use it.
Good luck.
