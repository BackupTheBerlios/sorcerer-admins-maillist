Good point.
I read about that also.
Apparently laziness is something that can be selected
as on or off when it comes to uClibc.
And while it does provide some benefits
I expect it is still not as good as having
the code directly linked in at compile time.
Also when using uClibc with ELF libc
we can also enjoy the benefit of laziness.
So in a way if X is laziness
and both glibc and uClibc's ld.so have it.
Then it can be divided out on both sides of the equation.
And consequently, uClibc still has the boon
of being considerably smaller when statically linked,
yet should have glibc commensurate start up times
when dynamically linked.
