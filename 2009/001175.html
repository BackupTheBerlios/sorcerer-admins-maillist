> Most services that can be launched by xinetd

> in order to activate the service.

>

>
> Maybe it's IANA's fault?
> Maybe it's software authors' fault by not following
> the service names listed by IANA?
> Take your pick.
> Since I neither specify what services are official
> nor write software that uses xinetd
> I know that it is not my fault.  :)
>
>
>>?shouldn't having the default configuration files at
>> least make sure definitions for their respecitve services exist at

>
> If I write the xinetd configuration file for the service
> then usually it does appear in /etc/services
> However, some of those I have not checked in years.
> They may be obsolete or been given a new designation by IANA.
>
> A box will automatically receive new /etc/services and /etc/protocols
> that are downloaded from IANA and stuffed into a tarball at
> the Sorcerer's website.
> However, boxes that do not have the spells
> "services" "protocols" installed
> will not have those files updated.
>
> Some SAs use custom /etc/services.
> They expect that service to port translation is slow
> and that /etc/services wastes memory.
>
> 536K 2009-05-02 19:49 /etc/services
>
> Indeed 536K for some 5000 service to port number translations
> is a bit excessive.
>
> If a SA wants a service to run on a specific port
> then a service name is not always required.
> The SA could write the xinetd service file using
> a port number instead.
> Or at least that used to work if it no longer does.
>
> The xinetd service files that we have control over
