Thanks to a kernel parameter
the SA can specify exactly how much RAM that kernel
should see and be able to use during boot.

The memory page size can be discovered with a libc call.
However, I forgot which one.  :)
It is in a manual page somewhere
or in the glibc online documentation.

In comparison to IA32 x86_64 will tend to waste memory.
However, the consumption might not be double.
Programmers are not going to waste effort to detect
the int size and specify u_int32_t for every int that they
would normally only require 32-bits for.
And the compiler does not know if the program requires
only 32-bit ints or 64-ints when a generic int or unsigned int is declared.
Consequently, anything int tends to require double the space
on x86_64 as it would on IA32.
However, strings, which are series of bytes or characters or 8-bit ints,
do not require more space on x86_64 than they would on IA32.

For decades I have considered that "int" was improperly designed.
Precise definitions should be used for memory size grouping.
bit = 1 bit
nibble = 4 bit
byte = 8 bits
word = 2 bytes
double word = 4 bytes
quad word = 8 bytes

Redefining the bit size of a word or bit size of an int
based upon the CPU architecture is confusing.
It is as bad as the little endian, big endian, pdp endian situation.

There are types defined in libc and glibc that may vary
and one would not know if those types specify an integer
of size 32 bits or of size 64 bits.
There is plenty of confusing stuff.
It is possible for a programmer accustomed to writing
programs on x86_64 architecture with glibc to write a
program that would not run properly on IA32 architecture
for lack of adding a line such as:

#define _FILE_OFFSET_BITS    64
