> I'm trying out the new Installer for x86_64 and I'm running into a problem.
> When I do an md5sum check on the source files, it looks like the entire
> directory /usr/src/linux fails.  When it tries to re-transfer the directory,
> I very quickly see
> "tar: usr/src/linux/..file name...  : Cannot stat: No such file or director
> "
>
> I can only see this for an instant before the next file wants to be
> re-transfered.  I've said yes well over 1000 times so far.  I actually just
> put a weight on the 'enter' key.
>
> I did an md5sum check on the .iso before I burned the image and it was good.
>  I haven't yet checked the sources md5sums because the first check isn't
> done yet.
>
> Any ideas?

Cute.  :)

The first time I attempted to generate the ISO I forgot to remove
the /usr/src/linux directory.
Then, after I did that I forgot to regenerate the initrd image.
Therefore, the initrd image expects that the files that were
in /usr/src/linux were installed when they do not exist on the I/R.
Best solution is probably to avoid doing the check, entirely.
Consider it broken in this release.
The next x86_64 IR I roll will not have that problem.
I should have used my macro script that invokes all my smaller scripts
in sequence.

The I/R should still, otherwise, be use-able.
Meanwhile, I am doing some prep-work and debugging of
a very advanced and different version of sorcery.
Yes, that does mean that it uses less RAM
has easier to read code, and is faster.
But there are several places where I have not removed
or rewritten the references to index directories that are no longer used.
So more grepping and editing is ahead.
I do not want to release it in a state that could frustrate SAs.

Therefore, if no show stopping bug is reported with the IRs
I will continue to hone the sleek new sorcery so that when
I do re-wrap the IRs the newer sorcery can go on them.
For now I added sorcerer-x86_64-20070905.iso9660.bugs
that explains the bug you reported.
Thanks for reporting the error.
