It might not know what your root is
You could try adding kernel parameter
root=/dev/....

Also it is very easy to cause the above problem if....
The kernel is compiled without directly compiling in support for:
the required IDE SCSI SATA PATA chipset drivers;
or the filesystem for which /

Yes, you compile a kernel with modular support for your
drive hardware chipsets and modular support for the file
systems that you have deployed.
But it will not boot.
That is because I detest initial booting using an initramfs or initrd image.
Because each linux kernel for Sorcerer boxes are custom compiled,
for near optimal performance and memory usage,
Sorcerer boxes do not require an initrd or initramfs image for loading
drive chipset driver modules and filesystem modules before mounting /
that resides on an actual disk or disk like device.

I am amazed you have a screenshot of it.
Were you using a virtual machine?

Do check that you have the support for the correct chipset
that the virtual machine emulates in addition to the chipset
of the actual machine if you are testing a kernel on the virtual
machine that you may want to use on the actual machine.
If, for example, you compiled in support for the chipsets
of the actual machine thinking, yes, this is correct,
but then run it on the virtual machine which requires different
drive chipset then of course it will not be able to access the virtual drive.
