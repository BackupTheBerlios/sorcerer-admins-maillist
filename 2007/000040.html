Perhaps it is?
I rewrote the two lines which apply the patch file.
It now uses what is expected to be the precise path/name to the patch file.

PCX=1
while  [  -n  "${SOURCE[$PCX]}"  ];  do
  patch  -p0  <  $SOURCE_CACHE/$SPELL/$VERSION/${SOURCE[$PCX]}
  ((  PCX++  ))
done

That should also be faster than before.

Next time you cast the spell you could try inserting a
set -x
at the top of the build.
It will however, end up with many lines of cast execution also
dumped into the compile log.
But if you at least encompass the part where it begins
the patch to the part where it finishes the patch with:
set -x
...
...
set +x

Then, whatever it is doing that is unexpected shall be revealed.
:)

Also...
You could check out those files in the source cache.
Those two files that are supposed to be patch files?
Are they patch files?
Or are they some poorly configured server that wrote
Some type of error 404 message into a text or HTML file
instead of actually throwing an error 404 back to the browser.
When patch is invoked on a file that is not a patch file,
patch usually process it silently and produces no line
stating that any file was patched or if there was a problem
with the patch file.
That is most likely the problem that the files in the source
cache which are supposed to be patch files are not patch files.
Therefore, no matter how many times the spell casts the
source is not properly patched.
