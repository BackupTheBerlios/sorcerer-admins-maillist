By disabling demos and examples and
removing my previous ccache for qt-x11-opensource
it was able to compile within the 2G limit.
Therefore, I replaced the attribute "new" with "solo"
Maybe it is now good for use on ia32?
Also it will probably compile faster.  :)

However, updating it while using the previously
saved ccache cached object files might exceed the 2G limit.
I suppose I will find out on the next update.

Lastly, 2G might still not be enough on x86_64 platforms.
Many default data sizes are twice as large, so is some of the code.
If the 1.5x round up is roughly accurate I expect it will exceed 2G.

Anyone want to give it a try on x86_64, with the spell from svn,
and report how much space is required prior to installation.
Adding something like:

du -sm /usr/src/sorcery &&

on the line above prepare_install &&
aught to print the size required into the compile log.


Interesting comparisons:

The uncompressed ccache tarball for qt-x11-opensource is 230M !!!
For contrast the uncompressed firefox ccache tarball is merely 70M
For contrast the uncompressed qt-x11-free ccache is  a meager 31M
The qt-x11-free ccache probably does contain cached object files
for demos and examples while the qt-x11-opensource one does not.
Likewise the the archive caches for qt-x11-free and qt-x11-opensource
are also approximately a magnitude difference in size.

Makes me wonder.
I guess we shall see how it performs near the end of this year.
