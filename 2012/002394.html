Obviously, having processes such as updating installed software
defer input/output to user owned processes is desirable.
And that is what recent changes accomplish.
sorcery should now be able to provide smoother performance.

Last week I was using the "pv" program from
http://www.ivarch.com/programs/pv.shtml
in order to rate limit some I/O tasks.
However, rate limiting was not exactly what I wanted.

Now cock is used for rate limiting.
Unlike other rate limiting programs cock has an auto-mode.
With auto-mode full bore output is permitted
until an impending I/O spike is encountered.
Then I/O is deferred momentarily.
This allows the kernel more I/O for loading user processes,
swapping, and other tasks that compete with sorcery for disk I/O.

Additionally, cock allows rate limiting specified in megabytes per second.
And a box with the "min" feature enabled
will always rate limit I/O tasks to 1 megabyte per second
in order to minimize sorcery's utilization of resources.

SAs that want to specify a specific value for rate limiting
I/O tasks can do so using sorcery's configuration menu.
8 megabytes per second provides a very reasonable
compromise between performance and system spontaneity.

Like most aspects of sorcery
this is yet another
that can be complete ignored
while still receiving a benefit.

Installing and upgrading software requires processing power and I/O.
Yet both should be deferred when needed by user owned processes.
Sorcery should be capable of completing it's tasks
without a detectable loss in primary role performance.
