Well, a tiny bug exists in the sorcery spell.
It could cause wane to fork bomb.
Other than that it works decently.
However, if I was anyone except Kyle or Evert
then I might consider not updating
for another couple of days.  :)

The bug in the sorcery spell
was introduced to deal with the odd situation
of having a customized /etc/sorcery/config
that should not have been customized.
Evert reported the problem of having newer sorcery installed
without updating /etc/sorcery/config
Only possible conclusion was that I did not anticipate it
did not add backward compatibility for a function
that prevents customization of /etc/sorcery/config
No box should have that file customized.
All customizations involving sorcery go into
/etc/sorcery/local.config
It says that at the top of /etc/sorcery/config

I slipped the fix for sorcery into the spell after testing.
Therefore, the bug went forth in the daily grimoire,
since I accidentally typed : where a ; should be.
Unfortunately, instead of adding a solution for a situation
that should not exist
I instead introduced a distribution wide bug
which could cause update complications on boxes
that otherwise would have updated reasonably smoothly.
Sorry about that.

Ah it is irritating.
Also a bug might exist in immune-exam-transition.
Anyone who definitely wants to keep their source cache
aught to execute
# cp -al /var/cache/sources{,.save}
before attempting to update.
Some boxes properly transfer /var/cache/sources/ to /aux/can/source/
On other boxes the source cache evaporates.
The code looks fine.
So the mystery persists.

So aside from an evaporating source cache
and a few machines fork bombing wane
the update happened.
While definitely not a failure,
I am irritating myself over the aforementioned aspects.

BTW, sdelta3 update patches will be unavailable for a while.
And the master source cache will be less stocked than usual.
And as usual a month from now we can look back at this moment
and laugh heartily, although some people might already be laughing.  :)
