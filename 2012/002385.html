> Over a decade ago someone showed me something called an OSI chart
> that explained an operating system as 7 layers.
> http://en.wikipedia.org/wiki/OSI_model
> This is conceptually useful for neophytes that do not yet recognize
> the interconnectedness.
> However, the model is not useful for veterans that see and realize the
> connections.
>
> The majority of software acts as units that partially process and
> translate information for a separate layer.
> For example wayland can act as a layer that sits between the X server
> and the kernel.
> And in general the processing that wayland does makes X output slower
> than allowing X to communicate directly with the kernel.
> And the kernel provides and controls access to hardware.
> Not everyone would agree, but programming tends to become easier
> the further removed it is from interacting with hardware.
>
> kernel <-> glibc <-> coreutils <-> bash
>
> That is where coreutils is situated.
> And if the aforementioned hypothesis is true
> then kernel hacking is very challenging,
> glibc hacking is moderately challenging,
> coreutils programming is an average challenge,
> and bash programming is easy.
> However, exceptions to every rule exist.
> In my experience shell programming is very different
> than assembly, C, C++, Pascal, perl, PHP, Fortran, etc...
> Consequently, mastery of shell programming is not easily acquired.
> And master does not mean that one is capable of writing shell scripts,
> but rather that the shell scripts created run incredibly fast.
> It is the main reason why most people except that shell programs
> will run extremely slow when compared to an implementation in any
> other language.
>
