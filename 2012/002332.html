Sometimes for security related fixes I visit xorg's website

However, if someone had very good awareness of the situation
then that person could convey it to me and
I could also properly adjust the "secure version progress" for such spells.
As we all know spells have potential for an unlimited amount
of version progress, but most often have "stable" "unstable"
and "secure version progress."
"Secure version progress" is the choice that is best suited
for the SA that wants updates that fix security vulnerabilities
but in general does not want the upgrades that might introduce bugs.

The updates a software author makes may be for reasons of adding
new features and functionality which also incurs the highest potential
for introducing new bugs.
And some updates may be merely bug fix updates.
Sometimes the method which the software author changes
the version may indicate the difference.
This is evident with linux kernels, gnome software source tarballs,
and some other projects.
Although we have some potential for identifying this
and perhaps potential for putting the information to use;
discussion has not occurred on this topic yet.
Well, actually, it has, but on a spell specific basis,
such as the decision to try avoiding the
x.y.0 release of KDE or the x.y release of gcc
and then wait until the first bug fix release for the new branch.

The concept of an implementation that would
either promote stability or promote progress
would make a worth discussion.
Perhaps an implementation could allow the SA to select a preference
that would be applied in a somewhat generic manner
rather than having to specifically select version progress
for each installed spell?

Spells in the grimoire currently favor progression to newer stable releases.
I only hold back or regress the promotion of a new version
for a spell at a rate of about maybe 1 in 1000 updates.

I created version progress so that SAs could gain easy access
to installation of alternate release branches of software.

that release more than merely a stable source tarball

which I update when I see new versions appear.
I do this frequently for software hosted on GNOME's ftp server.

Version progress also helped eliminate disinformation
concerning whether Sorcerer was providing unstable software.
And the answer to that became YES,
but only when the SA selects "unstable version progress" for a spell.

And if pressing enter, then does it present a prompt to login?
If so then that is not too far from ideal.
Install KDE on it and it will gain a GUI login.

On the other hand, some kernel graphics modules
that have kernel mode switching capability
will entirely and permanently blank the screen when loaded.
The following modules are blacklisted in files located in /etc/modprobe.d/
ch7006, i915, nouveau, radeon, vmwgfx

If the hardware exists at boot and udev is coldplug loading
the module and it is problematic then the person at keyboard
will likely never reach a prompt for selecting installation type.
Therefore, this problem seems unlikely.
But if anyone knows of additional modules that should be blacklisted
then please convey.
BTW, blacklisting modules prevents udev from loading them,
however the X server can still load them.
Therefore, blacklisting graphics related modulesis not usually problematic,
but I only blacklist what I know causes problems
rather than blacklisting all graphics modules.

> You are telling me to fix the installation issue
> and this will take a long while for me.

Giving a verbose description of the problem might
solicit insightful replies on the email list.
Not to seem cruel but if the problem with the I/R
is located between the chair and keyboard
then typically a new I/R release
will not solve nor circumvent that problem.

However, if the problem exists in on the CD
then it might be fixed by burning a new ISO9660 image.
I always recommend that a SA test the quality of the burn
by doing a md5sum of the device containing the I/R
and comparing it with the md5sum file.
Or it can be dd(ed) back to a file and both files can be compared.
Currently, the I/R does not contain code to detect
and report physical media errors.
But the small size of the I/R helps mitigate the probability.

If the problem is distribution wide,
which means it could effect all boxes of similar hardware loadout
then identifying and circumventing this problem
would be the best course of action prior to generating new I/Rs.
A good example of this is when earlier versions of I/Rs
had a difficulty that only manifest
when the computer had a floppy disk drive attached.
Strange is it not?
I adjusted some init-scripts, rolled a newer I/R
and the problem was solved.
There was also at least 2 other methods that a SA could have
locally solved or circumvented the problem.
Therefore, it was an annoying, but not an insurmountable, problem.

Other distros release I/R CDs or floppies that are entirely unusable
and then do not admit the fault nor even provide fixes for more than a week.
In contrast when I release a new I/R
I tend to do 3 days of relaxation in front of the keyboard
so that I can receive and respond to errors
and if necessary re-commence the release cycle.
I have yet to do a new release and
then rush out on a 1 week long vacation.
But if I did then I would certainly announce
it and probably place that release in a beta/ directory
to indicate the quality is not known beyond my testing.

For example, I recently upgraded boost.
The new boost seems usable and compiles and installs fine.
However, some software projects require an older version of boost
and do not compile with the newer boost.
Therefore, a few spells that were otherwise fine before the upgrade
have since gained the "broke" attribute.
I am "on the wall" about this upgrade.
If I can pound the sources into usable shape to make
them compile and link with an older version of boost in /opt
then the problems creating by the boost upgrade can be ignored.
However, that was not accomplished on a first attempt.
I intend to make a second and more lengthy attempt at it eventually.

This type of upgrade demonstrates a condition
where opinions about the optimal course of action differ.
With boost as a noteworthy exception;
I can usually make such upgrades go smoothly
by providing a spell that installs an older version of software in /opt.
I anticipated a successful outcome with usual methods,
but have yet to achieve it.
This less than ideal upgrade outcome
could create cause for some SAs to prefer
that boost remain at the previously installed version.

# au progress boost custom 1_46_1
