Evert, why tell him about cpref files?  *chuckles*
Please, forget everything about cpref files and never use them.
They exist for enabling debug information and nothing else.  ;)

Yeah, go ahead and export that and laugh heartily
at the many software errors that will happen.

Most software authors do not know how to create Makefiles
with sufficient rules to allow parallel make to work flawlessly.
Some of the software projects will parallel make sporadically.
Some might fail the first time and then complete on a second
attempt when some of the object files are already cached by ccache.

Software projects that have demonstrated success with parallel make
are granted the makej trait in the spell.
Software projects that deserve optimization
are granted the optimize trait in the spell.
SAs are not expected to set environment variables:
CFLAGS; CXXFLAGS; LDFLAGS; MAKEOPTS.
Do so at one's own peril.

However, for the /+/base/ root file system I recommend
the feature base be enabled.
That way it can boot on similar yet different hardware
such as transition from AMD to Intel or Intel to AMD.
Other root file systems should not have the base feature enabled.
The base feature deprives CFLAGS and CXXFLAGS
of -march=native and instead gives it a specific
-march=something where something is the base architecture.

In general a SA aught to toggle features.
See output from:
# au help feature

The above command provides information about available features.

# au feature

The above shows the features that are currently enabled

# au feature $FEATURE

The above command toggles the state of the feature
which is substituted for $FEATURE
Consider features like light switches.
They are on or off.
No dimmers; no timers; no motion sensors!  *chuckles*

If wanting to restrict the amount of processors that sorcery utilizes
then set that using
# /etc/init.d/cgroup-init-root configure
Personally, I never allow sorcery the first processor
on any of the root file systems.
The above step must be done for all root file systems.

The root file system which is /media/root/main/ or /+/base/
can adjust how control groups are allocated for each root file system
by running:
# /etc/init.d/cgroup-init-rootfs configure

Perhaps the /+/base/ root file system should be granted all processors by
default,
however the /+/test/ root file system should only be granted one processor?
If so then that would be the place to do it.
After the amount of processors and/or memory
granted to a root file system is set.
Then the control groups within those root file systems
can not elevate the limits.
So if /+/test/ is granted the final processor or final two processors
then the sorcery in /+/test/ can not use all 8 processors,
but only the final two.

Even if sorcery is granted utilization of all the processors
then when in competition for processor cycles with user
and system owned processes
then sorcery utilities will utilize no more than 1/64th processing power.
So in other words, a person can update a workstation
without noticing a performance impact provided
that the workstation installed RAM is sufficient.

To speak more succinctly,
do not try to out tweak the utilities.
Their intellect is hard coded,
but adapts dynamically to the situation.
That is to say the amount of spells
that concurrently moil varies.
Likewise the utilities that produce heavy I/O
will operate sequentially with each other instead of concurrently.
The sorcery utilities will attempt to maximize utilization of resources
while at the same time defer resources if the person logged in
wants to play games or watch videos or other activities.
We do not drop to runlevel 1 for updates
which is typical of traditional POSIX.
Instead, updates on our root file systems
happen concurrently with the root file system
providing it's designated role.

Do consider that I deserve all the accolades for this achievement.
I heavily utilize the Linux kernel hacker's control group implementation.
It is far superior to POSIX nice.
And in that fashion sorcery can update a root file system
like the devil is chasing it if nothing else is competing;
or it can update utilizing nearly only idle cycles
when competing with primary role processes.

In this fashion the SA can focus on SA tasks.
Most of the micromanagement games have been eliminated.
Even the linux kernel configuration is provided
rather than the SA creating one.
It is a little bit less fun.
SAs aught to have the skill of Linux kernel configuration.
However, having excellent modular Linux kernel configuration files
is a grant convenience.
With that said, a custom Linux kernel configuration file
can be created and used instead.
Doing so is not recommended.  *chuckles*

>> Hi there, and welcome to Sorcerer.
>> augur does not have a manpage. Instead, just type "augur" on the
>> command line, and it will tel you it's options.
>> One of us might write a manpage someday...
>>
>> On your second question, the short answer is yes it does, but you
>> don't want to mess with it.
>>
>> Longer answer:
>> Because I know you want to mess with it, and it's your machine and
>> your right, I'll explain the mechanism to you, but if you blow up your
>> box, don't blame me.
>>
>> Sorcery will by default read:
>> /etc/sorcery/cpref/default
>>
>> If that file does not exist, sorcery will pick sensible defaults,
>> which you can fine-tune in the sorcery menu. You will basically get
>> optimization -O0 for most small packages, and -O2 for stuff that makes
>> sense to have optimized, and some esoteric flags for spells that needs
>> them.
>>
>> In the Sorcery menu, under "Toggle" sorcerer features, you can select
>> whether to compile for base architecture or not. This switches default
>> between -mtune=native and -march=native.
>> You can also switch on "lto" for link time optimizations. If you want,
>> switch on the complementary -Os for spells that have the "lto"
>> capability.
>>
>> Now, if that is not specific enough for you, and you want to specify a
>> default that is different from what I described above, here is how:
>>
>> # mkdir -p /etc/sorcery/cpref
>> # nano mtune-O2
>>
>> inside, put your CFLAGS, like so:
>>
>> export CFLAGS="$CFLAGS -march=native -O2 -pipe -fno-warnings "
>> export CXXFLAGS="$CFLAGS "
>>
>> Or whatever you may like.
>> To make sorcerer compile with those enabled,
>> # ln -s /etc/sorcery/cpref/mtune-O2 /etc/sorcery/cpref/default
>>
>> Please let me re-iterate that this changes the defaults to whatever
>> you put in that file, and can completely break your box. Also, it
>> probably won't have the desired effect of making everything faster. I
>> used to run with a default cpref pretty close to the one I described
>> until very recently. Now I don't, and let sorcery pick the cflags for
>> spells, and my desktop "feels" more responsive.
>>
>> However, there is some really useful application for when you want to
>> mess with just one spell's cflags:
>>
>> # augur cpref $SPELL <tab> <tab>   <- will list the files in
>> /etc/sorcery/cpref
>> # augur cpref $SPELL $CPREF_FILE    <- will set that spell to utilize
>> the cpref file that you make.
>> # augur cpref $SPELL  <- will clear the custom cpref for that file
>> # augur cpref    <- will just list the spells with custom cpref
>>
>> For instance, I have in i7 laptop. Current firefox segfaults when
>> -march=native is passed to it, because it gcc gets the avx features of
>> my CPU wrong. Just for firefox. strange, but I like compiling for my
>> cpu type, so now I have the following file
>> /etc/sorcery/cpref/firefox
>>
>> Contents of which:
>> export CFLAGS=" $CFLAGS -mno-avx "
>> export CXXFLAGS=" $CXXFLAGS -mno-avx "
>>
>> I issued the following command:
>> augur cpref firefox firefox
>>
>> And now just firefox is compiled with the added -mno-avx
>>
>> The same mechanism can be applied to any single spell that you want to
>> mess with, and limits the potential damage to the rest of your box.
>>
>> Have fun.
>>
