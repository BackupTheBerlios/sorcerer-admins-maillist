Ha, there goes Evert again sharing
the fancy lore with neophytes.
However, that is nearly correct.
The configuration file is called .config.custom
The file .config.stock is installed by the spell linux-config
Become confused, and linux-config will overwrite .config.stock
and destroy the custom configuration.

Providing that a SA can create
a kernel configuration that is usable
then it can be utilize that easily
by copying it to /usr/src/linux/.config.custom
However, with that said....
Do not do it!
*chuckles*

What if I start using oh something like unshared namespace.
And it happens so that unshared namespace is used.
Yet, a new SA might not realize that.
So the new SA creates a Linux configuration file
that is suitable for all the features that the SA
anticipates using and omits including the features
that this POSIX requires in order to function.
Guess what happens?
Failure to boot.

The best a SA could hope for is to start with a stock configuration
then modify it to create the custom configuration by
removing hardware drivers for hardware that is not installed.
Then the SA is responsible for doing that EVERY time
that a major Linux kernel is released.
Therefore, when transitioning from Linux 3.9.11 to 3.10
the SA must create an updated .config.custom
that is also compatible with the .config.stock
Is it a chore?
Well, yes it is.
I update the modular Linux kernel configurations
for every major update of Linux.
I previously provided 3.
Now I only provide 2.

Anyway, forget about /usr/src/linux/.config.custom
Almost nobody uses it.
I also do not use it.
The only people who would use it are also those
who would have some very different initramfs generation scripts
that enable the creation of micro-sized initramfs images,
like on the scale of 16M or less.
Those boxes fail to boot if anything goes unexpected
and they can not be rescued without a rescue label
or a rescue USB SSD.

BTW, this POSIX is not for use on legacy and embedded computers.
However, persistent people never stop trying.
For that reason the previous IA32 deployment disk
had software compiled with the 80486 instruction set
which is the least advanced IA32 platform which Linux
can be expected to boot on.

Still, the deployment disk probably would not boot on a computer
that has less than 1G of installed RAM.
But that would not stop someone
from making one that would
on a USB SSD if such was desired.
Bare minimum RAM for booting is probably around 256M.
Updating much software aside from what is provided on /+/base/
would not be possible with less than 512M or perhaps 1G of RAM.
Certain software projects would create serious swap thrashing
that could last for hours or longer while linking.

> >> >> Hi there, and welcome to Sorcerer.
> >> >> augur does not have a manpage. Instead, just type "augur" on the
> >> >> command line, and it will tel you it's options.
> >> >> One of us might write a manpage someday...
> >> >>
> >> >> On your second question, the short answer is yes it does, but you
> >> >> don't want to mess with it.
> >> >>
> >> >> Longer answer:
> >> >> Because I know you want to mess with it, and it's your machine and
> >> >> your right, I'll explain the mechanism to you, but if you blow up
> your
> >> >> box, don't blame me.
> >> >>
> >> >> Sorcery will by default read:
> >> >> /etc/sorcery/cpref/default
> >> >>
> >> >> If that file does not exist, sorcery will pick sensible defaults,
> >> >> which you can fine-tune in the sorcery menu. You will basically get
> >> >> optimization -O0 for most small packages, and -O2 for stuff that
> makes
> >> >> sense to have optimized, and some esoteric flags for spells that
> needs
> >> >> them.
> >> >>
> >> >> In the Sorcery menu, under "Toggle" sorcerer features, you can select
> >> >> whether to compile for base architecture or not. This switches
> default
> >> >> between -mtune=native and -march=native.
> >> >> You can also switch on "lto" for link time optimizations. If you
> want,
> >> >> switch on the complementary -Os for spells that have the "lto"
> >> >> capability.
> >> >>
> >> >> Now, if that is not specific enough for you, and you want to specify
> a
> >> >> default that is different from what I described above, here is how:
> >> >>
> >> >> # mkdir -p /etc/sorcery/cpref
> >> >> # nano mtune-O2
> >> >>
> >> >> inside, put your CFLAGS, like so:
> >> >>
> >> >> export CFLAGS="$CFLAGS -march=native -O2 -pipe -fno-warnings "
> >> >> export CXXFLAGS="$CFLAGS "
> >> >>
> >> >> Or whatever you may like.
> >> >> To make sorcerer compile with those enabled,
> >> >> # ln -s /etc/sorcery/cpref/mtune-O2 /etc/sorcery/cpref/default
> >> >>
> >> >> Please let me re-iterate that this changes the defaults to whatever
> >> >> you put in that file, and can completely break your box. Also, it
> >> >> probably won't have the desired effect of making everything faster. I
> >> >> used to run with a default cpref pretty close to the one I described
> >> >> until very recently. Now I don't, and let sorcery pick the cflags for
> >> >> spells, and my desktop "feels" more responsive.
> >> >>
> >> >> However, there is some really useful application for when you want to
> >> >> mess with just one spell's cflags:
> >> >>
> >> >> # augur cpref $SPELL <tab> <tab>   <- will list the files in
> >> >> /etc/sorcery/cpref
> >> >> # augur cpref $SPELL $CPREF_FILE    <- will set that spell to utilize
> >> >> the cpref file that you make.
> >> >> # augur cpref $SPELL  <- will clear the custom cpref for that file
> >> >> # augur cpref    <- will just list the spells with custom cpref
> >> >>
> >> >> For instance, I have in i7 laptop. Current firefox segfaults when
> >> >> -march=native is passed to it, because it gcc gets the avx features
> of
> >> >> my CPU wrong. Just for firefox. strange, but I like compiling for my
> >> >> cpu type, so now I have the following file
> >> >> /etc/sorcery/cpref/firefox
> >> >>
> >> >> Contents of which:
> >> >> export CFLAGS=" $CFLAGS -mno-avx "
> >> >> export CXXFLAGS=" $CXXFLAGS -mno-avx "
> >> >>
> >> >> I issued the following command:
> >> >> augur cpref firefox firefox
> >> >>
> >> >> And now just firefox is compiled with the added -mno-avx
> >> >>
> >> >> The same mechanism can be applied to any single spell that you want
> to
> >> >> mess with, and limits the potential damage to the rest of your box.
> >> >>
> >> >> Have fun.
