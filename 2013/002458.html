The deployment docent has a step in it
where the goal is to install code to the MBR.
It says what file to use, where it is located,
and the command to type to install have the file content
installed to the MBR.
And it has a test to verify that the content of the MBR is bootable code.

The only reason it might fail is if the command is typed incorrectly
or if the device is typed incorrectly or if the BIOS has enabled boot virus
protection
which will prohibit writing to the MBR and typically cause the computer to
beep
if anything attempts to write to the MBR of a disk device.

Most people do not have MBR protection or boot virus protection enabled in
the BIOS CMOS.
If dual booting with a commercial operating system then definitely enable
it after
installing code to the MBR.
I read in a news article today an expected 15,000 or more computers may be
infected with
an Internet banking virus in Japan that is capable of transmitting
usernames and passwords.
So apparently computer viruses are still something of a problem these days.
However, probably very few of them aim for installation in the MBR of a
disk.
Back in the days of MSDOS viruses that infected the MBR of disks,
especially floppies,
was a much more common occurrence.

One of the things the deployment docent does not describe is EFI booting.
Seem as if EFI booting might be one of those things that I can not support
until
I have a computer that is capable of doing EFI booting in order to play
with it.
Therefore, I am assuming that those who have it know how to make it work
or install a POSIX that is compatible for EFI booting.

A good example of how things are implemented in this POSIX
is acquired by contrasting the first x86_64 port with the multi-lib
implementation.
The x86_64 port was fine work contributed by Jan Merka.
Because sorcery at that time required a pre-loadable library to track
software installation
the feat was a magnitude more difficult to accomplish than it should have
been.
Running any ELF 32-bit program with the 64-bit ELF preloaded library caused
a segmentation fault.

In contrast the current multilib POSIX we deploy these days
is what occurred after I purchased an x86_64 computer.
I am not going to say how much better, but we probably all agree
that the mutlilib port is better smoother and than the initial x86_64 port.
And we can run win32 programs in wine or run pre-compiled ELF 32-bit
software
when the software authors do not provide source nor 64 bit ELFs.
Overall it seems like decent quality to me although it required a bit of
effort to achieve.

For that reason I expect that that someone could contribute the code and
spells
required for an EFI boot implementation for this POSIX.
I would happily accept the contribution if someone did.
However, typically the best bet for excellence is if I do it myself.
However, I am not buying a new computer.
My current work station with 4 cores, with 6 root file systems,
with disk efficiency that I previously expected was impossible;
it still exudes that new feel new computer feel, new computer smell, new
computer speed.
So I have no complaints or wants at the moment.
Oh, but it does not do have EFI firmware, oh well.

> > The deployment docent only describes the tools and methods
> > that are useful for achieving a GPT deployment.
> > However, anyone who has legacy partitions already has
> > a bootable MBR installed and already has a partitioned fixed disk.
> >
> > If the second fixed disk is to be used entirely by modern magic
> > and if the bootloader that chains to modern magic does so by
> > loading and running the code in the MBR of the disk containing modern
> magic
> > then no reason exists for not transitioning that disk to GPT.
> >
> > However, if legacy partition tables are desired
> > then simply use the fdisk program provided by util-linux.
> > I made no attempt to document the legacy partitioning tools.
> > However, I also made not attempt to remove from the from the I/Rs
> > in order to conserve space.
> > As I previously said, deploying the 2013 I/R is not unlike driving
> > a F22 air superiority aircraft 4 blocks to the grocery store.
> > It is over-loaded or over over-kill for the meager task of deployment.
> >
> > My computer boots by having the BIOS load the MBR from the first fixed
> disk.
> > I can switch it to have the BIOS load the MBR from other disks or SSDs
> > if I want or CDROM.
> > Anything works fine.
> > However, if someone is dual booting and therefore using a 3rd party
> > boot management program
> > in order to select an operating system to boot
> > then the best choice is to have that program be programmed to chain to
> > the MBR
> > of the disk that contains Modern Magic 2013 if it exists on it's own
> disk.
> > If it does not exist on it's own disk
> > then the bootloader must chain to the code in the boot block of the
> > partition designated
> > to hold /boot/ for the POSIX.
> >
> > As Evert previously mentioned the boot block will be updated.
> > However, the MBR for the POSIX is not updated.
> > The MBR's content is set at deployment time based upon what
> > the SA writes to it or whatever existed the from a previous deployment.
> > I always recommend installing fresh code to the MBR.
> > That way one will know that that the MBR is not empty,
> > nor contains a virus from a previous operating system,
> > nor contains some odd grubby, lilo, or otherwise unexpected bootloader
> > program that was installed to the MBR instead of the boot block, etc...
> > The installed syslinux on the I/R provides image files suitable for
> > both legacy and for GPT.
> > However, it does not provide a single MBR code file that is suitable
> > for both.
> > So the code in the MBR must be appropriate for the type of partition
> table.
> >
> > Legacy MBR code can not boot GPT partitions.
> > GPT MBR code can not boot legacy partitions.
> > The only requirement for making it work
> > is that the code in the MBR matches the partition table type.
> > BIOS age should be irrelevant.
> >
> > Both GTP and legacy partitions work equally fine in a HVM qemu virtual
> > machine
> > as long as the code installed to the MBR of the virtual disk is
> compatible.
> >
> > In order to avoid that length explanation and any confusion concerning it
> > then deployment docent only provides documentation that describes
> > the use of the tools and methods for doing GPT partitioning.
> > Simply ignore that and instead use the tools installed by util-linux
> > and use the pre-existing MBR or the legacy MBR provided by the syslinux
> > project.
> > ms-sys is no longer provided on the I/R
> > and so ms-sys can not be used to install bootable MBR code.
> > I removed ms-sys from the I/R because ms-sys
> > was only useful for legacy partitions and had not been updated
> > with code suitable for booting GPT disks.
> > Therefore, doing a deployment with legacy partitions
> > is only slightly less convenient than in 2012.
