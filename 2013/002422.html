Yeah, I dunno. I don't have a problem with the 14 docs you authored,
because they presumably explain that step in the process.

I endeavored at one point to develop a cookboot/tutorial install manual,
which preceded sentient and the auto-desktop install. It reached almost
60 screenshots when I scrapped it because the install had changed
fundamentally.

Considering there are plenty of virtual consoles to go around, perhaps
directing the installing person to open a separate virtual console and
running an app which provides dialogs with docs that the installing
person can switch back and forth between while following the docent on
another virtual console?

I'm just not sure about telling someone during an installation, where
they're on the machine itself with no other around, that they should
visit an URL or file in /usr/doc or especially, a man page.

I have no idea what this new docent installer is going to provide and
how well it will document the install process so that a user can follow
along and complete an install, so we'll just need to wait and see, but I
didn't personally find the curses dialogs explaining what needed to be
done arrogant at all.

There's a certain amount of explaining and default options/selections
that need to be present during an installation so that someone with
little exposure can run through it, making reasonable choices, and have
a functional system that boots when they're finished.

For example, w/o a complete tutorial treatment like I provide here:

http://northtech.us/content/20120213/easy-lvm-installing-linux-entirely-lvm-partitions

...saying that someone can set up LVM2 for all of their mount points
(including /boot and /) and pointing them to a man page or /usr/doc/
simply isn't going to be sufficient for even many average SAs out there
to complete without a bit of it spelled out for them.

Now, if there was an ncurses based dialog which could offer such
information for the installing person to run in a separate virtual
console, then the power to be informed would truly be at their
fingertips, along with examples of how to accomplish what they might
want to do.

But to say, check the man page. Well, that's just going to cause people
to end the install IMO.

We definitely need documentation specific to Sorcerer Linux. Much of the
documentation scattered about the Internet is ancient and irrelevant,
with commands like scry, rather than showing that it's parrt of auger;
or just what auger, cast, etc., do and a comprehensive set of docs which
describe their complete usage.

Right now, what we have are a lot of docs and guidance that is only
contained in the archives of this list. I think that this keeps many
from adopting Sorcerer, or leads them to giving up. I'm particularly
concerned that Sorcerer not become thought of as, "the Linux equivalent
of Minix, suitable for training but not considered for production server
environments.

> The person at the keyboard is responsible for learning and deciding
> the commands to type at the command prompt.

I think examples and suggestions for safe defaults should be present.

> Therefore, at the step involving creating file systems
> then the SA would read documentation about various file systems
> and manual pages which describe the available tools
> and finally the SA would type the commands to complete the step.

That's a pretty tall order I think. Sure, we're really only talking
about mke2fs, blkid, mkswap, swapon, tune2fs, e2label, mount, chroot,
and knowing about your fdisk tool of choice, but expecting noobs to
master concepts by referring them [primarily] to the man pages might end
up with them doing bizzare things they think they should, like:

mkfs -t ext4 -v -N 800000 /dev/sda4

...and totally confuse them when/if they are asked to generate their own
/etc/fstab, etc.

Some of those man pages kinda suck, from the perspective of finding out
what it is you need to do, when it is simple, as they serve to make the
installing person think that they need a whole bunch of arcane arguments
in order to get it to work.

Expecting someone to understand the basics of what they're going to need
to setup their partitions with fdisk and then offering a tool which
walks them through choosing the mount points (like earlier, perhaps
existing Sorcerer IRs), would be a step backwards, and like you say, the
more and most advanced methods are still available to them anyway.
