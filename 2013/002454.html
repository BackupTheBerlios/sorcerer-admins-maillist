The GPT tools can transition to a GPT style partition.
As far as I can tell the age of the BIOS does not determine
whether GPT works or not.
What maters is whether the master boot record contains code
that is capable of reading a GPT.
However, I do not recommend changing to GPT partitions,
because GUID Partition Tables require more space
than legacy partition tables.
Therefore, the existing partition on the disk would overlap the partition
table,
which would be problematic.
However, if the second disk will contain only a single Modern Magic POSIX
then changing it to GPT should be fine provided that the bootloader used is
GPT aware.

A boot partition aught to be 1G in size or 2G in size if planning something
exotic.
A dedicated swap partition should be about 1/2 the size of installed RAM
and perhaps larger if intending to save memory to swap, power down,
and then resume on the next boot.
Remembering that sleeping to disk only works when the kernel being loaded
is the same as the kernel which ran when the computer slept.
Therefore, never suspend to disk after linux-* updates.

In the past I would tell people to visit TLDP, The Linux Documentation
Project,
and keep reading the documentation there until gaining sufficient
understanding
to wield the menu driven installation program.
In the past I would reply,
"I provide a POSIX; I do not provide a class nor computer based training in
POSIX administration."

However, I have changed my mind about all of that.
People were hungry for knowledge, begging for me to teach,
and so I created the deployment docent which integrates
the learning and the accomplishment.
It is even better than a class because in most cases I have omitted
from the documentation portions where I described computing
in the 1970s 1980s, 1990s, etc.. and tried to make what I write
both succinct and currently applicable.

Consequently, the appropriate response to questions such as
where root file systems are created is:
"Please follow the deployment docent because it explains
where this POSIX is programmed to recognize root file systems
and the files utilized during boot for mounting of file systems.

Modern Magic 2013 is a multi-root POSIX.
Yes, that means exactly what it suggestions.
A computer can have more than 1 root file system.
In order to accommodate multiple root file systems
certain semi-complicated changes were made.
It might not seem intuitive to all people,
but it continues a trend that was started before.
The deployment docent documentation describes it.

2013 Modern Magic has a much different feel to it than 2012.
Log files formats differ.
Log file locations differ.
Storage location for source tarballs differ.
Archive tarballs no longer exist.
Storage location for cache tarballs differ.

Reading the documentation elaborates on those differences.
And if dedicated file systems of sufficient size are created
for the portions of the file system that would store temporary,
long term, operating system data, and user home data
then enhanced performance can be achieved while fragmentation
over time can be mitigated.

I know Evert has something rather interesting concerning partitioning
and perhaps the use of loop backed file systems.
As for myself I have some of the dedicated storage and temporary partitions
shared among several root file systems.
That way I gain the enhanced performance benefit
without ineffectively utilizing available disk space.
The documentation describes how to accomplish this using mount binds.
That way each root file system for example can have it's own
unique /tmp/ directory that is backed by the exact same dedicated file
system
that is providing /tmp/ for all root file systems.

About 40 or so of the pages I typed were probably specific to this POSIX.
Some of it describes an entirely new support file system hierarchy unique
to Modern Magic because Modern Magic no longer uses places such as
/usr/src/sorcery/ /var/state/sorcery/ /var/cache/sources/
/var/cache/ccache/ etc...
The previously designated locations within the root file system hierarchy
were never ideal nor was it designed in mind for a source based POSIX.
The Modern Magic specific file system utilization
can be far more ideal for achieving and sustaining performance.
Read the documentation will tell what it is called where it exists
the roles of it's subdirs and more.

A person who deploys Modern Magic 2013
will want to read all of that documentation
otherwise they will be looking in places of the file system
no longer utilized and be,  "What, nothing is here, so where are the source
tarballs?"
Then they might encounter the POSIX specific hierarchy and either
not know what it is or maybe they might not notice it at all.
Or maybe they will see it and think,
"Well I run no programs from that location so I have no use for it
so I will go ahead and delete it."
Then they rm -fr it and sorcery loses all of it's information concerning
installed software projects.
Oops, what a mistake to make!
Now the person must re-deploy because they just obliterated something
important.

In contrast a person who reads all about it by following
the steps and documentation suggested by the deployment docent
should achieve a much better understanding and ability to administrate it.

For what it is worth the transition could have been even worse.
Instead we are utilizing a more advanced implementation of System V style
booting.
Imagine the potential for confusion if switching to upstart or systemd
or some other type of currently developmental init/shutdown/reboot
implementation?

Consequently, I tried to keep 2013 as close as practical to 2012
while still achieving the new functionality.
But many important changes exist.
Anyone who skips reading the documentation
will almost certainly sow the seeds
that grow into a moment of future regret.
