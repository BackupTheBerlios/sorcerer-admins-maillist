> The deployment docent only describes the tools and methods 
> that are useful for achieving a GPT deployment. 
> However, anyone who has legacy partitions already has 
> a bootable MBR installed and already has a partitioned fixed disk. 
> 
> If the second fixed disk is to be used entirely by modern magic 
> and if the bootloader that chains to modern magic does so by 
> loading and running the code in the MBR of the disk containing modern magic 
> then no reason exists for not transitioning that disk to GPT. 
> 
> However, if legacy partition tables are desired 
> then simply use the fdisk program provided by util-linux. 
> I made no attempt to document the legacy partitioning tools. 
> However, I also made not attempt to remove from the from the I/Rs 
> in order to conserve space. 
> As I previously said, deploying the 2013 I/R is not unlike driving 
> a F22 air superiority aircraft 4 blocks to the grocery store. 
> It is over-loaded or over over-kill for the meager task of deployment. 
> 
> My computer boots by having the BIOS load the MBR from the first fixed disk. 
> I can switch it to have the BIOS load the MBR from other disks or SSDs 
> if I want or CDROM. 
> Anything works fine. 
> However, if someone is dual booting and therefore using a 3rd party 
> boot management program 
> in order to select an operating system to boot 
> then the best choice is to have that program be programmed to chain to 
> the MBR 
> of the disk that contains Modern Magic 2013 if it exists on it's own disk. 
> If it does not exist on it's own disk 
> then the bootloader must chain to the code in the boot block of the 
> partition designated 
> to hold /boot/ for the POSIX. 
> 
> As Evert previously mentioned the boot block will be updated. 
> However, the MBR for the POSIX is not updated. 
> The MBR's content is set at deployment time based upon what 
> the SA writes to it or whatever existed the from a previous deployment. 
> I always recommend installing fresh code to the MBR. 
> That way one will know that that the MBR is not empty, 
> nor contains a virus from a previous operating system, 
> nor contains some odd grubby, lilo, or otherwise unexpected bootloader 
> program that was installed to the MBR instead of the boot block, etc... 
> The installed syslinux on the I/R provides image files suitable for 
> both legacy and for GPT. 
> However, it does not provide a single MBR code file that is suitable 
> for both. 
> So the code in the MBR must be appropriate for the type of partition table. 
> 
> Legacy MBR code can not boot GPT partitions. 
> GPT MBR code can not boot legacy partitions. 
> The only requirement for making it work 
> is that the code in the MBR matches the partition table type. 
> BIOS age should be irrelevant. 
> 
> Both GTP and legacy partitions work equally fine in a HVM qemu virtual 
> machine 
> as long as the code installed to the MBR of the virtual disk is compatible. 
> 
> In order to avoid that length explanation and any confusion concerning it 
> then deployment docent only provides documentation that describes 
> the use of the tools and methods for doing GPT partitioning. 
> Simply ignore that and instead use the tools installed by util-linux 
> and use the pre-existing MBR or the legacy MBR provided by the syslinux 
> project. 
> ms-sys is no longer provided on the I/R 
> and so ms-sys can not be used to install bootable MBR code. 
> I removed ms-sys from the I/R because ms-sys 
> was only useful for legacy partitions and had not been updated 
> with code suitable for booting GPT disks. 
> Therefore, doing a deployment with legacy partitions 
> is only slightly less convenient than in 2012. 
>
