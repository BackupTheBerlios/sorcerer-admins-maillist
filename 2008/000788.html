On Thu, Sep 25, 2008 at 6:32 AM, Kyle Sallee <kyle.sallee at gmail.com> wrote:
> Previous reports indicate that networks configured automatically by
> dhcpcd, dhclient and wpa_supplicant should not be started during sysinit,
> because they might not work as expected.
> Consequently, the networking templates provided by the networks spell
> for those templates have a modified start function that begin with:
>
> start(){
>  if ! [[ -d /opt ]]; then return 1; fi
>  if ! touch /etc/resolv.conf 2>/dev/null; then return 1; fi
>
>
> This way if when the templates run during sysinit
> they return 1 and networking configuration is delayed
> until networks invokes the templates again during runlevel.
>
> For the benefit of SAs with boxes as described above
> those lines can be inserted into already completed
> and moved templates.
> But if the box does not have a problem starting networks during sysinit
> then it probably does not require that solution.
>
> Note that the update of the networks spell will not change existing
> configured networks.
> I will not try to modify completed templates automatically.
> I want to avoid accidentally creating a problem
> that could break network configuration on already deployed boxes.
>
> This change is in response to emails about how to start networks during
> runlevel instead of sysinit and how to know when to start networks.
