Both of you have excellent points.
That is why the "sold" functionality exists.
"sold" is short for "secure hold"
It will hold installed software as long as the
installed version is not a security risk
and does not have to be recompiled in order
to acclimate to library updates.

# augur installed | while read; do augur sold $REPLY; done

Puts an entire box on sold if no installed spells
are already on sold, otherwise it toggles spell's sold state.

As for downgrading an installed spell...
There is:

# dispel --unsafe $SPELL
# cd /
# 7za x -so /var/cache/archive/$SPELL-......tar.7z | tar -xf -

Due to changes in logs made in /etc/sorcery/log/
and other changes in spell formats
I removed the
# dispel --downgrade
parameter to avoid granting SAs
the opportunity of loading their box with old software
where cast can not detect and fix ELF and LA errors.
It may not be a welcome change,
but it was the correct choice
for best ensuring that Sorcerer boxes
continue to function as expected.


When SAs make command decisions
to not update installed software,
such as the linux kernel,
yet to continue to update other installed software
then any problem that comes of it is a problem
that is caused by the SA and not the distribution maintainers.

Not rebooting into newer linux kernels
is nearly the same as having linux on hold.
Consequently, none of the security benefits
from having the newer linux installed are achieved
if the box never reboots into the newer linux.

If looking to assign blame to someone other than SAs
why not blame the coreutils software authors for
modifying their source to use features found only
in kernels newer than linux 1.0

Since the available versions of linux and linux-modular
in the Sorcerer grimoire are compatible with
the available versions of coreutils
this bug is not a distribution wide problem.

Better to request that the coreutils software authors
adjust their source code than to expect that distribution
maintainers would modify source code that they
did not write that already compiles and executes
properly on a box that is being properly updated.

I understand and acknowledge
becoming upset and inconvenienced
by this change to coreutils source code
that was made entirely by coreutils software authors.
A probability exists that if enough people
voice a contrary opinion on the coreutils email list
then those software authors will include a check
in the next release to decide whether to compile
using legacy code or code only compatible with
boxes running newer linux kernels.
That is the best solution,
because coreutils software authors understand
their source code better than anyone else.
That is why this discussion belong on the coreutils email list.


Often I am amazed that free software and
updates of free software work as good as they do.
Baking a distro is not as easy
making one's own custom pizza.

For those who want their pizza without anchovies
there is now a no_selinux spell.
Before casting it I would recast
libselinux just to ensure that libselinux logs are correct
so that removal of libselinux will not cause unexpected problems.

Upon casting no_selinux expect glibc coreutils
and many other installed software to re-compile and re-install.
Also expect that no_selinux is a one way ride.
If no selinux components installed is desired
and selinux will never be used
then casting no_selinux is an easy way to
exile, dispel, appropriate spells while
properly re-compiling, re-linking, and re-installing
effected installed software.

Lastly, the no_selinux spell has never been tested.
It may not accomplish the desired effect.
I recommend only using it on developmental boxes and
then reporting both success and bugs to the email list.

selinux support will still remain half cocked for I/R development.
Full support for selinux is no further along than
it was about a month ago.
Instead I wrote the no_selinux spell and finalized the
2009 format for the I/R and rolled some for your pleasure.

If anyone is feeling total displeasure then try
the freshly available Sorcerer I/Rs and marvel
their more compact design and faster and easier installation.
Today, is not a rainy day on every part of the globe.
