ok you requested for a more complet spec about progress support and 
repositories....

here is what I propose :
each spell have a stable and unstable progress, thoses progress return 
version values that will then be used by leech
I propose that the spell can provide a repository() function, this 
function shall be ran into the directory where we want the repository to 
be leeched, this function handle all the steps for making a compilable 
tree (ie create files if needed, at return of the function, you should 
be able to run build() in the tree without noticing that it's a 
repository tree)

# repository update : update the repository (there is sources in the dir)
# repository checkout : checkout a whole new tree (there are no sources 
in the dir)

leeching from a repository should be activated by an
# augur progress repository
if no repository function in the spell, then we shall fallback to stable 
version (not unstable) and if possible a warning shall be issued when 
doing the augur progress....

when a spell is leeched, leech shall try in order if there is no user 
progress specified for the spell :
- stable
- unstable
- repository

if repository shall be leeched :
- check if there are some source cache, 
(/var/cache/source/spell/repository/spell-date.tar.gz)
- if yes : uncompress the tarball into some leech place, pushd into this 
place, do repository update
- if no  : create the dir, do repository checkout


when an update is done :
no version information shall be checked, if the spell is cast 10 times 
in a row, the source shall be updated 10 times in a row....
