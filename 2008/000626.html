I agree with Jan about porting spells from contrib grimoires.

Also when porting spells...
Please move all url functions to directly above the
source function that it requires.


Example from xorg-server:

       url $XORG_URL
    source xserver/xorg-server-$VERSION.tar.bz2
       url $SOURCEFORGE_URL
    source mesa3d/MesaLib-$MVERSION.tar.bz2



Also url can be omitted:

The above can be written as:

    source $XORG_URL xserver/xorg-server-$VERSION.tar.bz2
    source $SOURCEFORGE_URL mesa3d/MesaLib-$MVERSION.tar.bz2


In that example the URL proceeds the filename separated only by a space.

Also a simple common sense syntax
is supported as shown in the ccache spell:

    source http://samba.org/ftp/ccache/ccache-$VERSION.tar.gz


In that example the source contains the full protocol::path/filename

No longer think of URL as a persistent array.
It is not.

I was considering removing support for url() entirely.
However, chaffing the URL using source() during indexing is a bit slower,
and I did not have an itch to again edit 3000+ spells for such a small change.

The new grimoire and sorcery will become available soon.
Yes you will be des-stroyed! by how much faster it is!

These changes are not just about performance.
I am checking many spells and fixing bugs and providing
the design we will require to easily support a grimoire 4x the current size.
Best to make these changes now when the grimoire is a meager 3000+ spells.

Because google is still not fixing their broken download servers
I am adjusting leech so that it no longer requests
nor requires headers before attempting to download files.
This will probably make curl stall indefinitely
when attempting to download from some broken servers.
We shall see.

leech should be faster with fewer invocations of find.
Indexing will be about 2x as fast.
cast and dispel will become a bit faster.

sorcery photon will be nearly fast enough
to escape the massive gravity of /dev/null
for yet another year.
