Actions that use a repository use the REPOSITORY variable.
$REPOSITORY is independent of stable(), unstable(), secure()
They are more different than apples and oranges.

For now for a spell to use a repository
then that spell must have the REPOSITORY=( ... ... ... ) variable.

Current REPOSITORY support is for cvs and svn.
I suppose git would work someone wrote the proper
commands inside the REPOSITORY variable definition.

Take for example, the REPOSITORY variable for ffmpeg:

REPOSITORY=( ""
             "svn co -r {$VERSION} svn://svn.mplayerhq.hu/ffmpeg/trunk ffmpeg"
             "svn up -r {$VERSION} svn://svn.mplayerhq.hu/ffmpeg/trunk" )

As you can see leech merely executes the proper commands.
It does not have those commands hard coded.

If this should work differently
then I require someone to write a verbose draft
explaining how it should work.

What you suggested thus far is merely 10%
The other 90% is still in the shadows unseen.
I have no idea how it will work.
As for the adding a progress repository command, great.
Yes, but what server?
What shell commands to access it?
So much is still missing.
It may yet be better to modify such spells
rather than to write an extensive overhaul
making the definition of spells more complex.

However, if the draft is good and the benefits
far outweigh the complexity
then a change would be a good solution.


For now REPOSITORY might seem like a quick hack.
And maybe it is:

# grep REPOSITORY *
ffmpeg:REPOSITORY=( ""
talloc:REPOSITORY=( ""

Only 2 spells in the official grimoire use it.
Yet CPU cycles are spent each time a spell is leeched
in order to check if REPOSITORY is definied.

I would expect that we require at least 8 or maybe 16 spells
that use REPOSITORY before we gain a clear idea of what
will be required to adequately support this functionality in leech.
Functionality for just 2 spells I am amazed if it works at all.  :)

The idea seems good.
But it requires yet more definition before we sculpt it into sorcery.
