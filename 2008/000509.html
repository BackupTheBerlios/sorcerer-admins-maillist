I have listened to feedback about the init-script changes.
Not everyone is pleased.
Neither am I.

The original design for cast handling of init-scripts
was overly elaborate, overly complex, and less than intuitive.
Therefore, I excised it.

I continued to ponder the original purpose of the changes,
"Have cast ask fewer questions about init-scripts."
How about instead,
"Have cast ask no questions about init-scripts?"

Instead cast will install init-scripts from
$GRIMOIRE/$SPELL.d/init.d.on  as mode 700 and
$GRIMOIRE/$SPELL.d/init.d       as mode 600.

What used to be optional will not be installed mode 600
so that it does not run automatically at boot and shutdown.
What used to be mandatory is now installed mode 700
so that it will automatically run at boot and shutdown.
Makes sense?

Here is the part that you will like.
The sorcery tools will remember the modes
that you assign to init-scripts.
init-scripts are still not customizable.
However, their permission mode is!
Want an init-script to not run at boot?
chmod 600 it.
The sorcery tools will remember and
preserve that mode across updates.
Want an optional init-script on?
chmod 700 it.
sorcery will keep it on across updates.

Very intuitive, yes?

The modes already set for init-scripts
by the overly elaborate method will stick,
because the init-scripts in /etc/init.d/
already have those modes.

Yes, I will preserve custom modes for mandatory init-scripts.
Turning off an init-script that is
essential for the box to properly boot is possible.
Please be careful.

For those who may not be reading this far
nor intend to remember it.
I will be adding instructions regarding init-scripts
to /etc/init.d/README
SAs can read README to know exactly what will work
and what will not work when modifying init-scripts.

Short docs right where they are needed most makes
for a better solution than remember the ever changing
methods of sorcery, right?

Hope everyone is having a fine week thus far.
