Configuration files for xinetd launched services can be
enabled and disabled by modifying the configuration file.
According to man xinetd.conf:

       disable          This is boolean "yes" or "no".  This  will  result  in
                        the  service being disabled and not starting.  See the
                        DISABLE flag description.


xinetd.conf also supports the attributes: disabled; enabled.
They can be specified to disable or enable lists of services.
Using disabled and enabled attributes is another method
for controlling available services without customizing
spell installed files in /etc/xinetd.d/

Achieving desirable performance from xinetd launched services
requires reading man xinetd.conf,
because xinetd is a feature rich replacement that
exceeds the ancient inetd in most every way.
And since xinetd supports multiple methods for
enabling and disabling services
cast will no longer ask questions about installing files in /etc/xinetd.d/
Spells with $SPELL.d/xinetd.d/*
will have all of the xinetd.d configuration files installed.

Feel free to modify files in /etc/xinetd.d/
Since they are configuration files
modifications will persist across updates.

Since most Sorcerer boxes have plenty of RAM for running
daemons that can respond to inbound traffic with minimal delay
the use of xinetd to launch services is declining.
xinetd still provides an excellent method for controlling
inbound connections that may meet and/or exceed what
can be accomplished by using methods such as tcp_wrappers, iptables.
For infrequently used services xinetd is worth the overhead.

However, mostly older software projects
provide services that can be launched from xinetd or inetd.
Fewer newer projects are not supporting xinetd and inetd.
This trend is not recent.
And it is yet another reason for it minimizing the sorcery utils
involvement in handling xinetd configuration files.
This eliminates some of the specialized code used for
merely for 15 spells out of a possible 3300 spells.

When new spells are written and/or old spells are updated
and a choice exists between launching services from init-scripts or xinetd
then init-scripts will be the only method provided.

If software is installed that uses xinetd to launch services,
please double check that files in /etc/xinetd.d/ are correct after
updating software that uses xinetd to launch services.
