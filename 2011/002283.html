One might want to ask the question...
What if the domain and exception information
fails to load into the kernel?
Does tomoyo entomb the box like selinux would?

Fortunately, tomoyo is not at all like that.
tomoyo starts permissive.
First a domain must be created, put into enforcement mode, and entered.
Then access becomes restricted according to that domain's rules
combined with applicable exceptions.

Making one's box unusable with selinux is more certain eventuality
than a misfire with a semi-automatic pistol.
The odds of making one's box unusable with tomoyo
is less likely than grounding a lightening strike while playing golf.

Does using tomoyo create more inconvenience than security?
Yes it certainly does if the implementation of domains makes it that way.
However, the implementation on Sorcerer
is constantly improving with respect to security
and the inconvenience is no greater
than running an additional program
when a wanting to change domains.

Sorcerer development maintains a balanced compromise
between functionality, speed, and complexity.
Tomoyo overhead is negligible,
domain transition speed is fast as a person can type,
and complexity is minimal.

I am trying to further minimize complexity by designing
the "limited" domain as an adequate box for most services.
That eliminates the overhead of having to update custom domains
when software that installs a service updates.
