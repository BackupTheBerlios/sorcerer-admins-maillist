While testing udev version 168 and saw the message:

udev[690]: error: runtime directory '/run/udev' not writeable, for now
falling back to '/dev/.udev'


What is a /run directory?
Both
http://www.pathname.com/fhs/pub/fhs-2.3.html
and
http://refspecs.linux-foundation.org/LSB_4.1.0/LSB-Core-generic/LSB-Core-generic/book1.html
have nothing about /run

Who wants a /run directory?
Just udev developers?
I found this explanation:

http://lists.fedoraproject.org/pipermail/devel/2011-March/150031.html

RedHat, SuSE, Debian and Ubuntu
all suffer from this same problem as described in the email.

However, since Sorcerer's init design is not a copy
Sorcerer has no problem with providing /var/run/ and /var/lock/
early in boot and then moving those file systems
to their proper location on the real root file system later.
As expected Sorcerer is not effected
by simple design flaw that effects major distributions.

I want to ignore the /run/ directory nonsense,
because it creates pointless additional work for me.
There is no benefit to /run/ on Sorcerer.
And udev still has a fallback method
for using a dot directory within /dev/
and therefore does not absolutely require /run/ in order to run.

However, now that udev is expecting a /run/
then other software vendors might also expect a /run/
was implemented for udev.
Consequently, this creates a falling domino effect.
udev runs early during init and is an important component
for plug and pray on modern POSIX boxes.
Therefore, udev has considerable mass.

The introduction /run/ on Sorcerer neither solves nor creates problems.
Sorcerer does not always precisely follow LSB and FSH guidelines.
When something is worth doing then Sorcerer does it regardless.
For the moment LSB and FSH documentation says nothing about /run/
Therefore, Sorcerer will implement /run/ merely because udev expects /run/.
And having /run/ might avoid future problems from software vendors
that expect /run/ was implemented for udev.
