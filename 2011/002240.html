# /etc/init.d/cpufreq configure     # <-- for configuration
# /etc/init.d/udevsettle configure # <-- for changing the timeout
# /etc/init.d/initdefault configure # <-- for changing the default runlevel.
# /etc/init.d/tmp configure # <-- for changing the size of tmpfs mounted on /tmp
# /etc/init.d/sysrq configure # <-- replaces /etc/init.d/sak and for
allowing full configuration of the sysrq mode.

Most of the above changes are in realization of tasks
that SAs previously accomplished
by copying and customizing the init-script
while making the original init-script non executable.
While I am not against customizable init-scripts,
configurable init-scripts seems far better.

/etc/init.d/initdefault might seem unnecessary.
However, it gives the explanation of the run levels
and the menu driven selection of a default runlevel
mitigates the possibility of the SA creating a typo in /etc/inittab.

Also many init-scripts are gaining support for a "status" parameter
or gaining improved information for a status request.

I nearly improved /etc/init.d/cgroup.
But I was unable to decide if I want to allow the SA to be able
to input a different mount point and select subsystems.
Does anyone have complaints about cgroups's subsystems
or mounting it on /dev/cgroup ?

Another question is...
Does anyone use gpm?
It has potential for use on a console,
but loss of functionality could result
by configuring xorg to use gpm's repeater.
Does the gpm spell require improvement or retirement?
I wonder this because I have not used it for a decade.
