Unfortunately, the Required-Start: Should-Start:
glues almost all init-scripts together
more solid than a ball of rice.
The failure of /etc/init.d/dev-cp
could cause the failure of nearly everything that comes after
if a dev-cp is required to run successfully.

However, sometimes problems happen.
For some odd reason / could be missing from /etc/fstab
There is an init-script that checks for a root filesystem
being defined in /etc/fstab.
If / is not defined in /etc/fstab
then a menu during booting appears that allows
the SA to select a device to mount for /
Could checking for that omission
and then circumventing the problem
create a more robust boot method?
In this situation the overhead for running
one additional init-script can be mitigated
by the potential good it can do by circumventing
a series of problems that would start with
the failure of being able to mount a real root filesystem.

If booting is intended to be more robust
then should not init-scripts be introduced
that anticipate, fix or circumvent problems?
Would not more init-scripts running be a better method
than excluding the execution 90% of init-scripts
when a problem happens early with udev?

Should the purpose of init revision be
added control, complexity and overhead?
Or should the goal be to increase robustness
so that problems which can be anticipated
can also be handled so that booting will
still be successful?
