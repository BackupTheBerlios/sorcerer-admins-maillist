Over the years both sorcery and the grimoire format changed.
Current grimoires are not compatible with ancient versions of sorcery.
Therefore, if the grimoire is updated manually
on an installation over a year old
then so must the sorcery be manually updated.
Even then the log files might not be entirely compatible.
Updates that jump more than 6 months are not supported.
Please install from current Sorcerer I/R
and update the box no less than quarterly.

Unlike distributions that have very rigid formats for packages
sorcery is a distribution that can change nearly and rule in order
to achieve faster or better performance.
For example...
In the past variables were declared in spells using
VAR=something which is very standard in bash.
However, in recent years variables are declared using functions.
This gives the advantage of providing faster indexing speed
and also the ability to override spell variable assignments so
that the SA can set the version for a spell to a non stable version
progress such as "latest" or "unstable" or "legacy" or even
to specify an exact version to use.

Newer version of the grimoire define variables using function calls
and those functions are defined in sorcery.
Therefore, old sorcery will not have the required functions
to index and work with a newer grimoire.

The grimoire and sorcery are obviously closely linked,
however they arrive in separate tarballs for the practicality
of being able to supply a stable and tested sorcery for
Sorcerer boxes while still enabling some test boxes to
use newer and more advanced sorcery.

However, in general the best sorcery to use with a grimoire
is the version of sorcery specified as stable in the sorcery spell
that is provided inside that grimoire.
Older sorcery might be able to cast some spells,
but the results would likely be incomplete and have problems
such as missing init scripts and/or missing configuration files.

So my advice is to not even try to cast spells
from a current grimoire until the installed sorcery is also current.
