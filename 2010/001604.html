> KDE is a lot of junk that I'm not using,

Now that is an interesting observation.
To make a point about how the new KDE-v4
compilation method is superior and customizable
I created the very fancy kdeedu-v4 spell
The spell has a huge amount of switches that can be used
to disable the compilation and installation of many
sub-components of kdeedu that might not be of interest.
kig = KDE interactive geometry
khangman = a word/phrase guessing game.
kalgebra = x = y + z
Really, I do not use any of that.
I only use the kiten Japanese/English dictionary.
So I have most of kdeedu-v4 disabled
and therefore it tends to compile quickly
and not require much disk space.

Something similar can be done for nearly all kde*-v4 spells.
But no SAs have expressed an interest in creating
very slim and precise KDE installations.

KDE has a lot of junk in the trunk,
but that does not prevent the SA
from deciding to put it on a diet to achieve a slim installation.
In order to rework the other KDE spells we only need SAs
to express some interest in having it done.
Otherwise I will be allocating time to improving other things.

> attach a KDE thing to the menu - at which point, it becomes easier to

There was a discussion about that.
Installing a standard something.desktop file in
/usr/share/applications/
will cause the software to show up on menus.
firefox is a good example of that.
those *.desktop files is about as easy as copy and edit.

However having KDE applications appear on other
desktop environment menus is not so easy.
Partly, because KDE installs in /opt/kde4/
and partly since it would not use the */share/applications/
directory in a flat manner, but also would have subdirectories.
