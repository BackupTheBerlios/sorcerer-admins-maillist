Sorry for the delay.
The culprit appears to be zlib.
Please do not install zlib version 1.2.4 on boxes
and revert boxes with it installed to version 1.2.3
Today's LAG=0 grimoire goes out with zlib at version 1.2.3
After undoing the zlib update firefox is behaving as it should.

zlib installs both a static link ar library /usr/lib/libz.a and ELF libraries.
However, most software only links with zlib's ELF library.
Therefore, for all boxes that were not entirely rebuilt follow the
last update of zlib
then merely installing the old version will fix problems like
segfaulting with other installed software.
That is because almost everything that requires zlib links with zlib's
ELF library,
which means they receive the immediate boons or banes of zlib being updates,
rather than that software also having to be recompiled.

Sorry for the inconvenience.
I have not hunted down the exact line or lines of code in zlib
where the bug is occurring.
Merely going back to the previous version seems to solve the problems.

Therefore, for all default LAG=1 boxes
updating should be about as safe as it normally is
in about 2 days from now, probably Tuesday or Wednesday
depending upon where a box is located.  :)
