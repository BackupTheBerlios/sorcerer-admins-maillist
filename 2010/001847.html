On Thursday, July 22, 2010, Kyle Sallee wrote:
> So do we go with the devil we know?
> Or do we risk new bugs
> by fixing what is not broken?

Well, my preference is for the system to boot as best as it can. Precluding 
services from starting due to a sometimes unimportant error would often result 
in more troubles than it is trying to solve, especially is certain essential 
services did not start (udev, dbus, etc.).

I would look more favorably on a compromise solution where only selected 
services would not start if there were errors issued by its required services. 
And for each such case, I would like to see a discussion whether this would be 
really needed. To summarize, I am OK if only relatively few services did not 
get executed due to errors of other init.d scripts, i.e. the default state of 
init.d scripts would be to run regardless.
