>> A box that is installed from the x86_64 I/R
>> and boots with that kernel will return
>> x86_64 for
>> $ uname -m
>>
>> A box installed from the Sorcerer IA32 I/R
>> will return i586 or i686
>> for uname -m depending upon the architecture.
>> It however, will not return x86_64 for uname -m
>> because the running kernel is not 64 bit capable.
>>
>> If uname -m does not return x86_64
>> and the box has an x86_64 CPU
>> and you want the additional headaches
>> additional RAM consumption and
>> slight speed increases of x86_64
>> then the correct solution is to install
>> Sorcerer from the x86_64 I/R and not the IA32 I/R
>>
>> If a person already knows to migrate from
>> IA32 to x86_64 without asking questions
>> then go ahead and do it.
>> Most SAs would not be able to accomplish the task.
>> As for me I would lack the time to accomplish it.
>> Since I already created a port of Sorcerer to x86_64,
>> thanks to help from Jan Merka,
>> I would simply install from the x86_64 I/R.
>>
>> Do not expect that that the BASH spell can be edited
>> so that it sets MACHTYPE to x86_64-pc-linux-gnu
>> and that automatically makes it an x86_64 installation.
>> No that just causes additional problems in the grimoire
>> for spells that use $MACHTYPE as a fast way to
>> detect if a box is IA32 or x86_64 in order to properly
>> adjust a spell.
>>
>> Setting MACHTYPE incorrectly creates problems.
>> The bash spells is doing the correct thing
>> by the method of how it sets MACHTYPE.
>> The only reason that it sets MACHTYPE in bash
>> instead of using what is already set
>> is because the MACHTYPE for IA32 installs
>> begins as i586-pc-linux-gnu
>> After bash casts it changes to i686-pc-linux-gnu
>> Does that change anything about the way code is
>> optimized for the installed CPU?
>> In almost all situations it changes nothing
>> except for very few sources
>> that disregard CFLAGS and CXXFLAGS
>> and attempt to set their own based upon $MACHTYPE
