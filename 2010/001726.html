If I could generate the error
I would first try to determine where the error is coming from.
If it happens during an init-script
then discovering which init-script should not be problematic.
Running of the init-scripts are logged.
init happens in two parts.
First, is sysinit
Final, is runlevel.
Their log files are in
/etc/init.d/log.d/

I prefer to cat them and view them with color
However, piping through:

sed 's:.\[[0-9][0-9]m::g'

will strip out the color codes.

cat $file | sed 's:.\[[0-9][0-9]m::g' | less

That command will allow one to pan around in it.
Virtual consoles can be scrolled up an down
with shift-page-up and shift-page-down
They are not back-buffered in RAM, so
switching to a different virtual console
will lose what scrolled above the 80x25
viewport on the previous virtual console.

If the error is not in an init-script
then best to figure out where the error
is coming from.

As far as I know there are no bugs on the new i/r
and therefore I recommend using the new i/r
Old I/Rs are collectors items and should be
placed on a dust where they accumulate and
trap lots of dust and never again enter into
a CDROM drive.
Please verify that the I/R being used to install is 20100529.

A reasonably fast way to circumvent the old error
that no longer exists on the current I/R
would be to enter the BIOS/CMOS configuration menu
during boot and disable the floppy drive controller.
No floppies and no problems I suspect.
I hope that works.

If it works then trash your Sorcerer installation.
It would probably be too old to update without complications.
Finally, download a new I/R that I rolled a few months ago.
Burn the I/R and install that Sorcerer.  :)
